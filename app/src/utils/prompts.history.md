# История промптов Cerebro

> Архив старых версий ключевых промптов. Текущие версии — в `prompts.ts`.
> Здесь сохраняется как они эволюционировали и почему меняли.
> Полезно когда: задаёшься вопросом "а зачем мы так сейчас пишем", думаешь о новых изменениях, или анализируешь почему один подход не сработал.

---

## System prompt (openai.ts)

### v1 — initial (2026-05-24)
```
You are an expert business strategist. Output ONLY raw JSON objects, one per line, no other text, no markdown. Always respond in Russian language — all titles, descriptions, and tags must be in Russian.
```
**Проблема:** "Expert business strategist" — слово-якорь в training data модели связано с McKinsey/HBR/VC pitch decks. Активировал академический жаргон на все 18 этапов. Локальные правила в промптах не могли с этим бороться — system prompt сильнее.

### v2 — current (2026-05-26)
```
Ты — опытный продакт-менеджер. Объясняешь коллеге за чашкой кофе.

Твоя аудитория — владелец малого бизнеса. Он не читает HBR. Он не использует слова "парадигма", "категориальный сдвиг", "трансфигурация", "ценностный континуум". Он говорит конкретно: деньги, люди, время, проблема, заявка, доплата, риск.

Твой главный страх — что собеседник через 10 секунд скажет "не понял". Поэтому ты пишешь так, чтобы понял с первого раза. Если используешь специальный термин — сразу объясняешь в скобках.
```
**Почему изменили:** Корневая правка тона на всех 18 стейджах одновременно. Сменили роль с "expert strategist" на "продакт объясняющий коллеге за кофе" — это автоматически downshift-ит академический тон. Дополнили запретом конкретных слов-маркеров жаргона ("парадигма", "категориальный сдвиг") и явным указанием на аудиторию (владелец малого бизнеса, не HBR-читатель).

## 4.1 Hypothesis Generation

### v1 — initial single-format (2026-05-25)
```
Stage: Hypothesis Generation
Turn the selected opportunities into testable business hypotheses.
Format: "We believe [specific customer] will [specific action] because [insight], resulting in [business outcome]."
Each must be falsifiable.
derived_from: ID opportunity spaces.
```
**Проблема:** Слишком общий. Модель имитировала pitch deck стиль. Метрики (Новизна / Сила / Реализация / Проверяемость) всегда 7-8 — нет калибровочных якорей, нет давления на честность.

### v2 — calibrated metrics + radical lineage awareness (2026-05-26 утро)
Добавлены якоря для каждого балла метрик (1-2: банальщина, 9-10: прорыв и т.д.). Добавлен баланс между variants/reframes. Calibration через explanation цены ошибки ("завышенная оценка = пользователь тратит недели на слабую идею").
**Почему:** LLM завышают самооценку систематически. Якоря с примерами + апелляция к цене ошибки лучше force distribution.
**Проблема:** Формат всё ещё один — модель могла свалиться в жаргон ("обратимый жилой слой", "категориальный сдвиг").

### v3 — 10-полевой формат от пользователя (2026-05-26 день)
10 структурированных полей в description:
1. Plain title (без академизма)
2. Клиент (конкретный сегмент)
3. Ситуация (реальный момент жизни)
4. Боль / напряжение
5. Гипотеза (We believe X will Y because Z)
6. Возможный оффер (простыми словами)
7. Почему коммерчески важно
8. Как проверить дёшево
9. Что опровергнет
10. Объяснение простыми словами — финальный тест "если не получается просто — гипотеза мусор"

**Почему изменили:** Свободный формат позволял жаргону прятаться. Структурированный формат с обязательным "Объяснение простыми словами" в финале работает как hard constraint.
**Проблема:** Контекст из предыдущих этапов всё ещё заражал жаргоном — модель видела "категориальный сдвиг" в карточках 2.2 и копировала.

### v4 — current — Plain Context Compression (2026-05-26 вечер)
Добавлен Step 0 перед генерацией:
> Перечитай контекст. Найди 5-8 самых острых направлений. Перепиши каждое в plain язык: кто / ситуация / боль / поведение / деньги. Выкини жаргон. Если кусок невозможно перевести в plain — он был пустым жаргоном, не бери в работу.

Только после Step 0 — генерация гипотез на основе plain плана.

**Почему изменили:** Самоусиление жаргона через контекст — главная корневая причина. Step 0 разрывает цепочку: модель сначала вынуждена переписать контекст в plain, потом генерирует на основе этого.

## 2.2 Obligatory Reframing

### v1 — academic (2026-05-26 утро)
```
Stage: 2.2 OBLIGATORY REFRAMING — RADICAL (атака на ОБЯЗАТЕЛЬНЫЕ признаки)

Цель: породить **категориально новые** ответы на ту же job-to-be-done через радикальное переопределение сущности. Здесь мы **выходим за пределы** сущности — получаем не варианты её, а заменители / новые категории.

Это самый творческий этап. Именно так появились iPad (сломал "клавиатуру у компьютера"), Tesla ("двигатель внутреннего сгорания")...
```
**Проблема:** Эти инструкции сами были пропитаны жаргоном — "категориально новые", "выйти за пределы", "самый творческий этап". Модель читала это в промпте и **зеркалила в выводе**. 2.2 стала источником заражения остальных этапов через контекст.

### v2 — current — plain tone (2026-05-26 вечер)
```
Этап 2.2 — Идеи где сущность становится чем-то другим

Идея этапа: найти решения, в которых исходная сущность фактически заменяется на что-то другое — но та же жизненная задача решается. Это не "улучшения той же штуки" (это 2.1), а "другая штука вместо".

Несколько примеров из жизни:
- iPad — людям нужна была клавиатура для ввода. iPad показал что экранная замена достаточна для большинства задач.
- Tesla — людям нужно было ездить. Электрика заменила двигатель внутреннего сгорания.
...
```
**Почему изменили:** Корневая правка по диагнозу пользователя — *"мы сами загнали модель в жаргон через свои промпты"*. Заменили "категориально новые ответы" на "идеи где сущность становится чем-то другим". Примеры с iPad/Tesla — оставили, но переписали в форме "людям нужно было X — Y заменил Z" вместо "iPad сломал клавиатуру у компьютера".

## 3.2 Contradiction Finder → Friction Deepening

### v1 — bottom-up Contradiction Finder (2026-05-25)
```
Stage: Contradiction Finder
Surface the core tensions in this space. Format: "X requires Y but also requires NOT-Y." These contradictions are where breakthroughs live.
derived_from: ID точек трения и измерений из Fundamental Dimensions.
```
**Проблема:** Слишком общий. Модель "находила" обобщающие contradictions которых не существовало — выдумывала структурные tensions поверх frictions. По факту дублировала 3.1 Friction Map.

### v2 — strict bottom-up with formula (2026-05-26 утро)
Усиление: жёсткая формула "X requires Y AND NOT-Y", обязательные ссылки на 2-4 frictions, тест abstractability, примеры правильного vs неправильного уровня абстракции. 3-5 contradictions максимум.
**Проблема:** Всё равно склонна к выдумке обобщений. Логика bottom-up провоцирует "найти" общую tension где её нет.

### v3 — current — top-down Friction Deepening от пользователя (2026-05-26 день)
```
Stage: 3.2 FRICTION DEEPENING / STRUCTURAL TENSIONS

ЦЕЛЬ: НЕ генерировать новые контракции с нуля. БРАТЬ конкретные frictions из 3.1 и вскрывать структурную tension под каждой.

ВЫБОР FRICTIONS ДЛЯ УГЛУБЛЕНИЯ:
- Если в 3.1 есть карточки с ★ — углубляй ТОЛЬКО их
- Если ★ нет — выбери 3-5 самых острых сам

ДЛЯ КАЖДОЙ ВЫБРАННОЙ FRICTION ОТВЕТЬ НА 8 ВОПРОСОВ:
1. Friction summary
2. Who suffers
3. What the user wants
4. What the system requires
5. Structural contradiction (formula "X requires Y but X undermines Y")
6. Why this contradiction persists
7. What current solutions hide or oversimplify
8. Why this tension may be commercially important

Если у friction нет реальной structural tension — честно скажи "это операционная проблема", пропусти. Пустота лучше выдумки.
```
**Почему изменили:** Top-down per-friction подход дисциплинирует модель — для каждой friction есть её tension через 8 фиксированных вопросов. Невозможно выдумать обобщение которого нет. Меньше галлюцинации.

## FMT_RULES (общие правила для всех 18 этапов)

### v1 — minimal (2026-05-25)
Простые правила формата JSON, без специальных мер против жаргона.
**Проблема:** Никакой защиты от LLM-словоблудия.

### v2 — добавлено rule 9 "Простой язык" (2026-05-26 утро)
Базовое требование "пиши простым языком, объясняй термины в скобках".
**Проблема:** Слишком мягко. Модель игнорировала.

### v3 — current — жёсткое rule 9 с blacklist'ом (2026-05-26 вечер)
```
9. Простой язык — ЖЁСТКОЕ правило.

ЗАПРЕЩЁННЫЕ КОНСТРУКЦИИ:
- "обратимый жилой слой", "категориальный сдвиг", "средовой интерфейс"
- "ценностная инверсия", "опыт-как-сервис", "трансфигурация процесса"
- ...

PLAIN-LANGUAGE TEST:
- Прочитай вслух. Мог бы это сказать владельцу химчистки и он бы понял?

Конкретика вместо абстракции: деньги, сроки, смета, мастер, квартира...

Internal vs Output: В рассуждениях можешь думать абстрактно. Но текст карточки — обычный язык бизнеса.
```
**Почему изменили:** Конкретный blacklist даёт модели якоря "именно это нельзя". Plain-language test через "владелец химчистки понял бы?" — задействует конкретный mental образ. Internal vs Output — даёт право думать сложно но обязывает выдавать просто.

## Промежуточные правки (не самостоятельные версии)

### 2026-05-26 — Plain Context Compression идея
Источник: совместный анализ с пользователем + ChatGPT.
Концепция: перед каждой генерацией модель обязана сначала переписать контекст в plain. Реализовано пока inline в 4.1 (Step 0), но идея применима ко всем поздним этапам.

### 2026-05-26 — Per-stage режимы plain language
- Module 01-03 (анализ) — рабочие термины с переводом в скобках допустимы
- Module 04 (Hypothesis / Critic / Shortlist / Validation) — жёсткий plain mode

### 2026-05-26 — Commercial signal lens
Добавлен `→ Commercial signal:` блок в описания карточек на 5 этапах:
- 1.5 JTBD (воля платить)
- 1.7 Failure Modes (цена сбоя)
- 2.1 Variants (кто заплатит)
- 2.2 Obligatory Reframing (есть ли рынок новой категории)
- 3.3 Cross-field (revenue аналога)

### 2026-05-26 — Lineage tracking
Селектор `cardLineage(project, card)` — вычисляет hasRadicalAncestor через граф derivedFromIds. UI бейдж 🔥 на карточках с radical-цепочкой. Метка не подделывается промптом — объективное свойство графа.

### 2026-05-26 — Module 2.1 / 2.2 split
Раньше: один stage Feature Challenge атаковал только accidental.
Сейчас: 2.1 Variants (accidental) + 2.2 Obligatory Reframing (obligatory). Разделение даёт каждому свой фокус.

## Принципы которые сейчас работают (саммари)

1. **System prompt = роль "продакт объясняет коллеге"**, не "expert strategist"
2. **Чёрный список слов-триггеров** в FMT_RULES rule 9
3. **Plain Context Compression** перед поздними этапами (пока только 4.1)
4. **Per-stage режимы:** Module 01-03 — рабочие термины с раскрытием; Module 04 — жёсткий plain
5. **Internal vs Output:** думай сложно, выдавай просто
6. **Structured формат на финальных этапах** (4.1 — 10 полей)
7. **Top-down per-friction для 3.2** — не выдумывать обобщения, углублять конкретные
8. **Commercial signal** на 5 ключевых этапах
9. **Lineage 🔥** — объективная метка radical через граф связей
10. **Конкретные примеры в промптах** — образцы лучше абстрактных правил
