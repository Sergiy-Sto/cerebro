Симптом с прогона Кофейни: в 1.2 / 1.3 / 1.8 / 3.1 видны дубликаты карточек. Иногда точные дубли по контенту, иногда — разные карточки с одинаковым номером (#1 #1). 3.1 Карта трений показывала 42 карточки вместо ожидаемых 8-15. 1.3 Fundamental Dimensions — 29 карточек вместо ожидаемых 8-10.
Двухкомпонентный диагноз:
  • Number collision. В Workbench.tsx auto-all для каждой stage-итерации использовался let maxNum = 0. Если стейдж генерировался дважды (пауза/продолжение, ошибка восстановления, кэш браузера) — каждый запуск стартовал нумерацию с 1, отсюда #1 #1 в UI.
  • Слабая нормализация title в дедупе. Reducer ловил только exact match с trim() + toLowerCase(). Невидимые NBSP ( ), кривые кавычки (« » „"), разные тире (— – -), хвостовая пунктуация — всё это проскакивало как "разные" title.
Решение в store.ts (ADD_CARD reducer):
  • Number collision-proof. Номер теперь вычисляется при записи из текущего состояния проекта: stageMaxNum + 1. Локальный maxNum в onCard — больше не источник правды, reducer переопределяет. Даже если onCard вызывался из нескольких параллельных потоков с собственными счётчиками — номера остаются непрерывными.
  • Нормализация title: NBSP + множественные пробелы → один пробел; «»""„" → стандартные кавычки; — – → дефис; хвостовая пунктуация (точки/запятые/двоеточия) убирается.
  • Защита от пустых title — мусор от LLM (пустая строка после нормализации) не сохраняется.
Решение в prompts.ts (existingBlock): усиленное предупреждение для модели с конкретными паттернами:
  • same title + разное форматирование/punctuation → дубль
  • same первые 6-7 слов с разным концом → дубль ("Сбой: невидимая очередь из мобильных" vs "Сбой: невидимая очередь у бариста" — это дубль)
  • same core idea с синонимами → дубль
  • splitting one card into two с slight angle → still дубль
Экономический эффект: дубли — это лишние карточки в контексте для следующих стейджей (input × число потомков). Если убрать ~30% дублей на ранних стейджах — экономия ~10-20% input-токенов на поздних. На прогоне Кофейни 3.1 с 42 карточками вместо 13 ставила Module 04 в положение перерасхода context на >2×.

2026-05-26 · Cost log honesty
💰 Cost log стал считать стоимость на лету — исторические записи отражают актуальный прайс
Проблема: CostEntry хранил cost как статичное число, вычисленное на момент записи. После обновления pricing.ts (старые $1.50/$10 → актуальные $5/$30 для gpt-5.5) исторические записи продолжали показывать старые цены. Лог недооценивал реальную стоимость в ~2.5× — $4.29 в логе vs $11 в OpenAI Dashboard.
Решение: добавлена функция entryCost(entry) — пересчёт по актуальному PRICING на лету. totalCostToday, totalCostAll, breakdown теперь используют её. appendCostEntry больше не пишет cost (поле оставлено optional для совместимости). Любые изменения цен в pricing.ts теперь автоматически отражаются на исторических данных.
После фикса: $4.29 → $13.41 (фактический $11-13 из OpenAI). Переоценка ~5-10% — в пределах погрешности (остаточный гэп от неучтённого prompt caching, который OpenAI считает по 10% от стандарта).
Урок про экономику: при структуре методологии "длинная цепочка из 18 стейджей" стоимость на 90% состоит из input, не output. Карточки/output — копейки сверху. Поэтому будущая оптимизация бить должна по input (AI-summary в цепочке, фильтр контекста), а не по output (длина описаний, число карточек).
Поправка: output не совсем бесплатен — он становится input для следующих стейджей. Жирная карточка из ранних стейджей умножается по всей цепочке потомков. То есть AI-summary в цепочке контекста — не "бонус для экономии", а обязательный механизм при длинной цепочке.

приоритет: высокий · 2026-05-26 (после теста Opportunity Lens)
🎨 Креативный режим — отдельная архитектурная развилка (выбор на старте проекта)
Контекст: На тесте Кофейни обнаружено что методология системно заточена на pain-solving (функциональный режим). Креативные концепты (магнит, эстетика, ритуал, идентичность) рождаются из другой логики, не из аналитики болей. Opportunity Lens (текущая правка) — это двунаправленная функциональная аналитика. Креативный режим — это вообще другой пласт.
Гипотеза: архитектурно нужны два режима, выбираемых на старте проекта (radio button: Функциональный / Креативный), сохранение в project.mode.
Структура креативного режима (черновик):
  • Свой Module 01 — не Reality Mapping, а Vibe Scan: эстетика рядом с категорией / места, ритуалы, истории / эмоции / архетипы и культурные референсы
  • Общий структурный слой с функциональным режимом: 1.3 Fundamental Dimensions / 1.4 Obligatory vs Accidental / 2.1 Feature Challenge / 2.2 Obligatory Reframing — деконструкция конструктора нужна в обоих режимах
  • Свои линзы концепта: Place / Ritual / Identity / Aesthetic / Story / Social Trigger / Persona / Legend / Audience / Naming Hook / Object of Desire / Repeatable Habit / Photo Hook
  • Своя 4.3 — не Hypothesis Generation, а Concept Hypothesis Generation — концепты-магниты, не гипотезы про боль
  • Своя 4.4 — критика не "что сломается" + "не пресно ли", а "сильный ли образ", "устойчив ли как магнит", "не банальный ли"
