## Что подставляется в `[плейсхолдеры]`

В каждом промпте есть переменные которые подставляются автоматически:

- **`[ТЕМА ПРОЕКТА]`** — то что ты ввёл при создании проекта (frame + constraints + criteria). Например: "Открыть кофейню в Киеве с фокусом на утренний поток".
- **`[Карточки предыдущих этапов]`** — все карточки прошлых стейджей по правилам контекста (отфильтрованные, не discarded, с учётом per-stage exclusions для 4.4 и 4.5).
- **`[Уже сгенерированные карточки этого этапа]`** — для режима "Думай ещё" чтобы не повторять то что уже есть.

```
ПРАВИЛА ФОРМАТА КАРТОЧЕК:

1. Маркеры уверенности per-пункт. Каждый пункт в перечислениях помечай в квадратных скобках:

   - Если поле "почему сейчас / Pain reduced / Commercial signal" нечем заполнить — честно напиши "по этому углу не вижу содержательного" вместо общих фраз

   - Парафраз чужого пункта: если правда похоже на другой пункт — удали один

   - Если карточка пропадёт — потеряет ли пользователь что-то ценное? Если "не особо" — НЕ пиши.

4. Missing Angles Check. После основного списка добавь 1-2 строчки:

5. Формат перечисления: маркированный список через переносы строк (\n).

6. Линейная мысль (аналитический вывод, не перечисление) — развёрнутыми абзацами с конкретикой.

7. Запреты:

9. Простой язык — ЖЁСТКОЕ правило. Пиши так, чтобы владелец малого бизнеса понял за 10 секунд без переспрашивания.

### Блок B — FMT_RULES_METRICS (= FMT_RULES + метрики; используется в 4.3, 4.4, 4.5)

Это блок A выше + дополнительно:

```
ДОПОЛНИТЕЛЬНО: для каждой карточки укажи метрики и краткий анализ:
{"title": "...", "description": "...", "tags": ["..."], "derived_from": ["..."], "metrics": {"novelty": 8, "strength": 7, "feasibility": 6, "testability": 9}, "analysis": "Новизна 8: [почему]. Сила 7: [масштаб]. Реализация 6: [сложно]. Проверка 9: [как быстро]."}

ЧЕСТНОСТЬ ОЦЕНОК (КРИТИЧНО):

LLM имеют tendency завышать оценки своих идей. Не делай так.
Цена завышения: пользователь уйдёт тратить недели и тысячи долларов на тестирование посредственной идеи.
Объективная низкая оценка экономит ресурсы. Это твоя главная задача.

ЯКОРЯ ДЛЯ КАЖДОЙ МЕТРИКИ:

Новизна (насколько неочевидно):
- 9-10: Прорывная идея, я не видел её ни в одном продукте мира.
- 7-8: Заметно отличается от очевидного. Есть редкие игроки делающие похожее.
- 5-6: Логичный ход, есть несколько похожих решений на рынке.
- 3-4: Очевидное решение. Первая мысль любого продакта.
- 1-2: Банальщина. То что выдаёт LLM на "сгенерируй бизнес-идею".

Сила идеи (масштаб):
- 9-10: Может построить unicorn. Решает массовую болезненную потребность.
- 7-8: Сильный продуктовый ход, может стать категорией. Ясная воля платить.
- 5-6: Хорошая нишевая идея. Понятный customer, ограниченный рынок.
- 3-4: Малый рынок или слабая воля платить.
- 1-2: Решение не для рынка.

Реализация (как сложно построить):
- 9-10: MVP за неделю силами одного. Все технологии готовы.
- 7-8: 1-3 месяца с командой из 2-4. Нет R&D, нет регуляторного риска.
- 5-6: 6-12 месяцев + capital. Возможны технические сложности но решаемые.
- 3-4: Большое R&D, лицензирование. 1-2 года минимум.
- 1-2: Не реализуемо в обозримом будущем.

Проверяемость (как быстро понять работает ли):
- 9-10: Можно проверить за выходные через лендинг / опрос / no-code.
- 7-8: 2-4 недели тестового запуска для первого сигнала.
- 5-6: Несколько месяцев пилота. Нужен реальный продукт минимум.
- 3-4: Полгода+. Нужен полноценный запуск.
- 1-2: Годы. Поведение не воспроизводится в тесте.

SELF-CHECK перед записью:
1. Поставил бы те же оценки если бы платил за тестирование из своего кармана?
2. Если все 5 гипотез у тебя 7-8 — это calibration error. В реальности распределение всегда неравномерное.
3. Не бойся ставить 3-4 объективно слабым гипотезам.
4. Если бы эту идею ты вбил в Google и нашёл топ-10 результатов — Новизна максимум 4.
```

### Блок C — OPPORTUNITY_LENS (вторая ось мышления; используется в 1.4, 2.1, 2.2, 3.1, 3.2, 4.2, 4.3)

```
OPPORTUNITY LENS — вторая ось мышления (обязательная):

Параллельно с поиском "что не работает / лишнее / больное" — ищи opportunity creation:
новые сценарии использования, новые ритуалы, новые форматы, новые роли, новые контексты,
новые поводы купить, вернуться или рассказать другим.

Для каждого пункта спрашивай ДВА вопроса:
1. Pain — что это может уменьшить или решить?
2. Opportunity — какую новую возможность это может открыть?

Опорные примеры различения подходов:

— "Стол стоит на полу" (случайный признак)
  • Pain-подход: убрать ножки чтобы экономить место.
  • Opportunity-подход: стол появляется из стены, становится частью интерьера, превращает
    маленькую квартиру в трансформируемое пространство → открывает сценарий "квартира где
    функции появляются по требованию".

— "Кофейня привязана к помещению"
  • Pain-подход: снизить аренду / оптимизировать расстановку.
  • Opportunity-подход: кофейня как точка на маршруте — кофе у парка, у школы, у станции,
    pop-up в неожиданных местах → новые контексты, новые поводы купить, новая аудитория
    (собачники, утренние пробежки, школьные родители).

Мы НЕ диагносты. Мы ищем коммерческие возможности.
Анализ и декомпозиция — это сырьё, не цель.
Цель — собрать из этого сырья новую ценную сборку, ради которой клиент готов платить
(а не починить то что и так работает).
```

Цель: проверить, расширить и обогатить Observation Scan через РЕАЛЬНЫЙ поиск в интернете.

ТЫ ИМЕЕШЬ ДОСТУП К ВЕБ-ПОИСКУ (web_search tool). Используй его агрессивно.

ПРАВИЛА ПОИСКА:
- Максимум 12 поисковых запросов на эту сессию — это ЖЁСТКИЙ потолок, не пожелание.
  Считай свои запросы. Когда дошёл до 12 — СТОП.
  Лучше упустить 13-й факт чем потратить $0.10 на лишний запрос.
- Карточек на выходе: МАКСИМУМ 10, обычно 7-9. Один поиск ≠ одна карточка — несколько поисков могут собраться в одну линзу.
- Обязательные линзы (по каждой минимум 1-2 поиска):
  • СУЩЕСТВУЮЩИЕ РЕШЕНИЯ — какие приложения, сервисы, продукты уже работают
  • РЕАЛЬНЫЕ ПРОБЛЕМЫ ПОЛЬЗОВАТЕЛЕЙ — что в Reddit / форумах / отзывах
  • ЗАМЕНИТЕЛИ — что используют вместо основного решения
- Опциональные линзы (если релевантны теме):
  • Региональные особенности
  • Свежие тренды и нововведения
  • Failure stories и горький опыт
  • Нишевые и необычные варианты
  • Цифры, статистика, размер рынка

ЗАПРОСЫ — НА АНГЛИЙСКОМ (лучший охват). Конкретные, не общие:
- ❌ "renovation"
- ✅ "renovation budget overrun reddit 2026"

**Цель:** породить варианты сущности через трансформации случайных признаков. Здесь начинается креатив, но остаёмся в той же категории.

```
[ТЕМА ПРОЕКТА]

Reality Mapping контекст (особенно Obligatory vs Accidental):
[Карточки из Module 01]

[Anti-duplicate блок если есть существующие]

Stage: 2.1 FEATURE CHALLENGE — VARIANTS (атака на СЛУЧАЙНЫЕ признаки)

Цель: породить варианты сущности через серию трансформаций над её случайными (accidental) признаками. Здесь начинается креатив, но мы остаёмся в той же категории — это разновидности той же сущности, не новые сущности.

⚠️ ВАЖНО: challenge — это не только "убрать / упростить / починить". Случайные признаки могут быть точкой опоры для новой возможности, если их не убирать, а УСИЛИТЬ или ПЕРЕОСМЫСЛИТЬ.

Возьми из 1.4 ИМЕННО СЛУЧАЙНЫЕ признаки (то что историческая конвенция, а не суть). Для каждого задай ОБА вида вопросов:

A. Pain-вопросы (что починить):
- Что если УБРАТЬ это?
- Что если ИНВЕРТИРОВАТЬ (сделать наоборот)?
- Что если ЗАМЕНИТЬ на противоположное?
- Что если АВТОМАТИЗИРОВАТЬ?
- Что если ПЕРЕНЕСТИ в цифровую форму?
- Что если СДЕЛАТЬ ДО / ПОСЛЕ основного процесса?
- Что если ОТДАТЬ другой роли?
- Что если СДЕЛАТЬ ДЕШЕВЛЕ / БЫСТРЕЕ / ПРОЩЕ / ПРЕМИАЛЬНЕЕ?
- Что если СОВМЕСТИТЬ с другой сущностью?

B. Opportunity-вопросы (что новое открыть) — обязательны:
- Что если этот признак — на самом деле скрытый рычаг, который надо УСИЛИТЬ, а не убрать?
- Что нового становится возможным, если изменить или переосмыслить этот элемент?
- Какая новая аудитория появляется? Новый ритуал? Новый контекст? Новый повод купить / вернуться / рассказать?
- Какие случайные свойства потенциально дают новые возможности, если их раскрутить вместо того чтобы оптимизировать?

Опорный пример различения подходов (стол):
Случайный признак: "стол стоит на полу".
- Pain-подход → убрать ножки чтобы экономить место.
- Opportunity-подход → стол появляется из стены, становится частью интерьера, скрытое рабочее место, превращает маленькую квартиру в трансформируемое пространство. Это уже не "решили проблему места", а открыли сценарий "квартира где функции появляются по требованию".

Каждая карточка = один сильный challenge. До 10 карточек, минимума нет. Если реально находится 3 содержательных — пиши 3. Баланс по факту: если все 3 сильных оказались opportunity-driven, это валидно.

Структура карточки:
- title: "Variant: <тип трансформации> <что именно меняем>"
- description:
  • что меняем и как
  • почему это интересно
  • кто бы выиграл от этого варианта
  • Pain reduced: какую боль или неудобство это снимает? (Если ни одной — честно "ничего, это чисто opportunity-ход".)
  • Opportunity opened: какую новую возможность это открывает? Новый сценарий / ритуал / роль / контекст / повод купить, вернуться, поделиться. (Если ни одной — честно "ничего, это чисто pain-fix".)
  • → Commercial signal: кто конкретно за это заплатит? Сколько? Уже есть подобные продукты на рынке? Если рынка нет — почему?

Правила:
- Работай только со случайными признаками. Если меняешь обязательный признак — это в 2.2.
- Если идея похожа на известный продукт на рынке — иди глубже.

Этап 2.2 — Идеи где сущность становится чем-то другим

Идея этапа: найти решения, в которых исходная сущность фактически заменяется на что-то другое — но та же жизненная задача решается. Это не "улучшения той же штуки" (это 2.1), а "другая штука вместо".

⚠️ ДВЕ ОСИ радикального переосмысления (используй ОБЕ, не только первую):

Ось 1: "Что если обязательное на самом деле случайное?"
Берёшь обязательный признак из 1.4 → ставишь под сомнение → задача всё ещё решается, но другим способом, другой категорией.

Ось 2: "Что если случайное на самом деле — упущенный главный актив?"
Берёшь случайный признак который выглядит банальным и второстепенным → представляешь что вокруг ИМЕННО ЭТОГО признака строится весь продукт → это меняет тип сущности кардинально.
Пример: для кофейни "стены в помещении" — случайный признак (мы фокусируемся на кофе). Но что если стены = главный актив? Это уже не кофейня, а галерея с кофе. Или подписка "арт-витрина для местных художников + кафе при ней". Это категория-сдвиг через подъём случайного в статус главного.

Несколько примеров из жизни:
- iPad — людям нужна была клавиатура для ввода. iPad показал что экранная замена достаточна для большинства задач.
- Tesla — людям нужно было ездить. Электрика заменила двигатель внутреннего сгорания.
- Uber — людям нужно было такси. Заменили "владение машиной таксистом" на "приложение и водитель когда нужно".
- Airbnb — людям нужно где переночевать. Заменили гостиницу на чужую квартиру.
- Spotify — людям нужно слушать музыку. Заменили носитель на доступ.
- Starbucks — кофе был случайно-побочным к "третьему месту между домом и работой". Бывшее случайное (атмосфера, диван, wifi) стало главным активом — а кофе стал лишь поводом туда прийти.

Возьми из 1.4 признаки (и обязательные, и интересные случайные). Для каждого подумай:
- По оси 1: что если обязательного признака не будет — что нужно вместо? Какая категория? Кто покупатель?
- По оси 2: что если этот случайный признак поднять в ранг главного — каким становится продукт? Какая новая категория? Кто платит?

До 6 идей, минимума нет. Лучше 2 сильных reframe чем 6 натянутых. Если для темы только 1 категориально-новая идея — пиши 1.

Структура карточки:
- title: короткое название идеи простыми словами. Пример: "Не ремонт, а замена обстановки одним пакетом" (а не "Реверс пространственной парадигмы").
- description:
  • какая ось задействована: ось 1 или ось 2
  • какой признак исходной сущности здесь играет ключевую роль
  • какая жизненная задача (из 1.5) всё ещё решается
  • что это становится за категория — простыми словами
  • кто первый покупатель — конкретный сегмент, не "все люди"
  • если есть аналог из реальности (iPad / Tesla / Uber / Airbnb / Spotify / Netflix / Starbucks) — упомяни
  • → Commercial signal: есть ли уже похожая категория где люди реально платят? Кто первый покупатель? Сколько готов платить?

Правила:
- Это должна быть другая сущность, не вариант. Если можешь сказать "это всё ещё та же штука" — это в 2.1.
- Связь с жизненной задачей обязательна. Если нет — это просто фантазия.

- ~30% Opportunity-amplifying — берём случайное свойство сущности (1.4 accidental, 2.1 opportunity-side, 2.2 ось 2) и усиливаем его в актив. Пример паттерна: Starbucks — кофе был побочным к "третьему месту", атмосфера стала главным активом.

- ~30% New-category / new-format — выходим в новую категорию или формат через 2.2 Reframes или 4.1 Cross-field аналоги. Пример: iPad заменил клавиатуру; Spotify заменил носитель доступом.

- остаток — свободные ходы: cross-field неожиданные комбинации, гибриды, эксперименты с новым каналом / новой аудиторией.

ВАЖНО: если все гипотезы получаются Pain-solving — ты в ловушке функционалистского мышления. Остановись и заставь себя написать хотя бы 1-2 opportunity-amplifying.

═══════════════════════════════════════════════════════════
СТРУКТУРА КАЖДОЙ КАРТОЧКИ:
═══════════════════════════════════════════════════════════

JSON-поле title — короткое название гипотезы простыми словами. Без академизма.
✅ "Временный апгрейд квартиры без ремонта"
❌ "Обратимый жилой слой как ценностная инверсия"

JSON-поле description содержит секции (русские заголовки, без номеров):

Клиент: Кто именно? Конкретный сегмент: "арендаторы в съёмных однушках Москвы, 25-35 лет", не "люди".

Ситуация: В какой реальной жизненной ситуации этот клиент сталкивается с проблемой?

Боль: Что болит — деньги, время, страх, тревога, утрата контроля? Конкретно.

Возможное поведение клиента: Что конкретно клиент может сделать — заплатить / подписаться / попробовать / откликнуться / переключиться. Без "мы считаем". Просто: "Клиент откликнется на оффер X потому что у него болит Y".

Возможный оффер: Что можно предложить в простых словах. Пример: "Обновим комнату за 1 день без ремонта: свет, хранение, текстиль, декор."

Почему это коммерчески важно: Почему повлияет на деньги, конверсию, спрос, стоимость, риск?

Как проверить дёшево: Лендинг, интервью, fake-door, ручной сервис, реклама-тест, форма, опрос. Конкретно.

Что опровергнет: Какой результат покажет что гипотеза слабая? Конкретная цифра или паттерн.

Объяснение простыми словами: 2-3 нормальных предложения как будто рассказываешь другу. БЕЗ жаргона.

═══════════════════════════════════════════════════════════

⚠️ ВАЖНО — типичный косяк который НЕ ДОПУСКАЙ:
- НЕ пиши в description "1. Plain title", "2. Клиент" — это были labels из инструкции

## 4.4 Critic Pass

**Цель:** жёсткая критика каждой гипотезы по двум осям. Одна карточка критики = одна гипотеза.

```
[ТЕМА ПРОЕКТА]

Гипотезы для критики:
[Карточки гипотез из 4.3]

[Anti-duplicate блок если есть существующие]

Этап 4.4 — Критический разбор (две оси критики)

═══════════════════════════════════════════════════════════
🔴 ЖЁСТКИЕ ОГРАНИЧЕНИЯ КОЛИЧЕСТВА И СКОУПА (соблюдай строго):
═══════════════════════════════════════════════════════════

1. Критикуй ТОЛЬКО карточки из этапа 4.3 Hypothesis (type=hypothesis).
   В контексте есть также 4.2 Opportunity Tree — это НЕ гипотезы, это направления.
   Их НЕ критикуем. Также НЕ критикуем 4.1 Cross-field, 3.x, 2.x — они уже сыграли роль как сырьё.

2. 1 карточка критики = 1 гипотеза. Не расщепляй критику одной гипотезы на несколько карточек. Все 2 оси (A + B) и все 3 способа провалиться + 5 вопросов оси B + итоговая оценка ВНУТРИ ОДНОЙ карточки.

3. Количество карточек критики = количество гипотез из 4.3. 5 → 5. 6 → 6. 7 → 7.

4. derived_from обязателен: каждая карточка критики ссылается ровно на одну гипотезу из 4.3.

═══════════════════════════════════════════════════════════

Стань самым жёстким критиком каждой гипотезы. Критика идёт по ДВУМ осям одновременно.

ОСЬ A — КРИТИКА ПРОТИВ ПОЛОМКИ (классическая):

Найди 3 самых вероятных способа провалиться. Какие предположения должны быть верны чтобы это сработало. Где риски — рынок, конкуренты, технология, регулятор, единичная экономика. Беспощадно честно.

Особенность для гипотез из цепочки 2.2 (с маркером radical lineage):
- Их способы провалиться другие — там риск "никто не понимает что это такое" / "категории не существует на рынке" / "регулятор не догоняет"
- Не критикуй их обычными мерилами вроде "что такого делает существующий лидер" — они не должны делать как он.

ОСЬ B — КРИТИКА ПРОТИВ ПРЕСНОСТИ (обязательная, новая):

Эта ось критикует не "сломается ли", а "стоит ли вообще делать". Без неё все смелые гипотезы автоматически проиграют осторожным в шортлисте.

Вопросы оси B:

1. Это итерация существующего или открытие нового пространства?
   Если итерация — насколько большая? Если ещё одна "оптимизация очереди в кафе" — пометь честно как операционную, не как стратегическую.

2. Если бы это сделали — рынок бы это заметил?
   Или это исчезнет в шуме как "ещё одна опция в меню"? Заметность — не маркетинг, это вопрос про категориальную смелость хода.

3. Не слишком ли это похоже на "что любой средний консультант предложил бы"?
   Если каждый второй продакт на разогретой сессии Miro выдаст похожее — это слабый сигнал.

4. Какая асимметрия апсайда?
   Что если сработает — насколько большой может быть результат? Гипотезы с симметричным риск/упсайдом — неинтересны.

5. Pain-balance: не вся ли гипотеза про "снять боль" без opportunity-разворота?

ЯЗЫК — простой человеческий. Не "structural failure", а "что пойдёт не так". Не "epistemic gap", а "люди не понимают зачем им это".

Структура карточки критики:
- title: "Критика: <простое название гипотезы>"
- description:
  • Ось A (риски поломки): 3 способа провалиться, каждый с пояснением почему
  • Ось B (риск пресности):
    - категориальная смелость
    - заметность для рынка
    - "очевидность" гипотезы
    - асимметрия апсайда
    - pain-balance
  • Итоговая оценка двумя цифрами: риск-аккуратность (0-10) и категориальная смелость (0-10). Слишком осторожная = 9/2 (надёжно но скучно). Слишком сырая = 3/9. Идеал = 7/7.

derived_from: ID критикуемой гипотезы.

[+ Блок B: FMT_RULES_METRICS]
```
