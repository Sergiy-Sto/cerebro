# Проект — Creative Core Workbench (cerebro)

> Справочный документ проекта. Видение, стек, архитектура.
> Живые задачи и статус фич — в `ЖУРНАЛ.html`.
> Не добавлять сюда текущие задачи.

---

## 1. Суть и цель

### Проблема
LLM на запрос «сгенерируй бизнес-идеи» выдаёт банальщину. Если пропустить тему через структурированную цепочку мыслительных методов с человеком-куратором на каждом шаге — результат нетривиален. Но человек не может удержать в голове десятки карточек со связями — нужна внешняя память и навигация.

### Что строим
**Методологический MVP, не продукт.** Минимум интерфейса, которого хватает чтобы один человек прошёл всю цепочку этапов без когнитивного утопания. Цель — проверить работает ли методология на 2–3 реальных темах.

### Намеренные ограничения
Минимализм здесь — не лень, а сознательная дисциплина. Каждая лишняя фича = шум при проверке методологии.

- Нет бэкенда, авторизации, БД — всё в `localStorage`
- Нет красивостей — сдержанный функциональный UI
- Есть OpenAI API: streaming генерация карточек, JSON-парсинг

## 2. Технический стек

**Используем:**
- Vite + React + TypeScript + Tailwind CSS v4
- `localStorage` (ключ: `creative_core_workbench_v1`)
- `useReducer` для state (без Redux/Zustand)
- OpenAI Chat Completions API (streaming)
- GitHub Actions + FTP Deploy на Hostinger (`cerebro.lexion.pro`)
- `crypto.randomUUID()` для ID

**Не используем:**
- ❌ UI-библиотеки (shadcn / MUI / Chakra)
- ❌ State-management (Redux / Zustand)
- ❌ Бэкенд / БД
- ❌ Анимации, темы оформления

**Поддерживаемые AI модели** (в селекторе хедера):
- `gpt-5.5` (дефолт) · `gpt-5.4` · `gpt-5.4-mini`
- `gpt-4.1`
- `o4-mini` · `o3`

Для gpt-5.x и o-series параметр `temperature` не отправляется (модели его не принимают). Для o-series используется `max_completion_tokens`.

## 3. Методология: 18 этапов в 4 модулях

> **Архитектурное решение (2026-05-25):** методология декомпозирована в 4 модуля по 4-9 под-модулей. В левой колонке UI отображаются модули с раскрытием под-модулей. Каждый под-модуль = отдельный stage, отдельный API-вызов, отдельная карточка-результат. Связи между под-модулями — через `derivedFromIds` в карточках.

### Module 01 · Entity / Reality Mapping (9 под-модулей)
| # | Stage | Что делает |
|---|---|---|
| 1.1 | **Observation Scan** | Карта реальности из знаний модели (без поиска) — типы, форматы, проблемы, заменители, существующие решения по структурным линзам |
| 1.2 | **Search Scan** 🌐 | **Web search через OpenAI Responses API** — модель сама делает до 12 поисков по линзам, синтезирует найденное с источниками. Confidence: `search_snippet_supported` |
| 1.3 | **Fundamental Dimensions** | Онтологический рентген — разложение на фундаментальные измерения (для объекта/процесса/сервиса) |
| 1.4 | **Obligatory vs Accidental Features** | Разделение признаков: без чего перестаёт быть собой vs историческая конвенция. Критический мост к Module 02 |
| 1.5 | **Jobs-to-be-done** | Какую жизненную задачу решает (не функция, а job — практический/эмоциональный/социальный/идентичностный) |
| 1.6 | **Users / Actors** | Карта ролей и интересов: что хочет, чего боится, чем измеряет успех, где конфликтует |
| 1.7 | **Failure Modes** | Где ломается, раздражает, дорожает, даёт плохой результат |
| 1.8 | **Substitutes** | Чем заменяют когда не хотят/не могут использовать напрямую |
| 1.9 | **Boundary Cases** | Пограничные случаи — что почти является, но не совсем |

### Module 02 · Feature Challenge
| # | Stage | Что делает |
|---|---|---|
| 2.1 | **Feature Challenge** | Атака на случайные признаки через трансформации: убрать / инвертировать / заменить / автоматизировать / совместить |

### Module 03 · Friction & Opportunity (4 под-модуля)
| # | Stage | Что делает |
|---|---|---|
| 3.1 | Карта трений (Friction Map) | Конкретные болевые точки |
| 3.2 | Поиск противоречий (Contradiction Finder) | "X требует Y и одновременно NOT-Y" |
| 3.3 | Кросс-доменный перенос (Cross-field Transfer) | Аналогии из других бизнес-секторов |
| 3.4 | Дерево возможностей (Opportunity Tree) | Концентрированные направления |

### Module 04 · Synthesis (4 под-модуля)
| # | Stage | Что делает |
|---|---|---|
| 4.1 | Генерация гипотез (Hypothesis Generation) | Falsifiable бизнес-гипотезы с метриками |
| 4.2 | Критический разбор (Critic Pass) | Failure modes для каждой гипотезы |
| 4.3 | Шортлист | Топ-3-5 + dark horses |
| 4.4 | План валидации | Cheapest fastest tests |

### Архитектура контекста между под-модулями

Каждый под-модуль получает в промпте **все карточки предыдущих stages** (по `order`). Модель в JSON-выводе указывает `derived_from: ["id1", "id2"]` — какие именно карточки предыдущих этапов использовались как основа. Это образует граф связей (данные сохраняются, визуализация UI отложена).

## 3.1 Search Scan архитектура (детально)

> Один из ключевых блоков — реальные внешние данные критичны для качества всей методологии.

**Технология:** OpenAI **Responses API** + built-in `web_search` tool.
- Endpoint: `POST https://api.openai.com/v1/responses`
- Параметры: `{ model, input, tools: [{type: 'web_search'}], stream: true }`
- Streaming через SSE events: `response.output_text.delta`, `response.web_search_call.in_progress`, `response.output_text.annotation.added`, `response.completed`

**Режим:** Light Research (snippets only).
- 1 web_search вызов = 1 запрос → 5-10 превью разных сайтов (title + snippet + URL)
- Модель НЕ переходит на сами сайты, работает только со сниппетами
- В отличие от Deep Research (Perplexity Pro / OpenAI deep-research) — там модель читает полный контент 100+ сайтов, стоит $5-20 за сессию

**Лимиты:**
- **Максимум 12 поисковых запросов на сессию** (страховка через промпт)
- **Обязательные линзы** (1-2 поиска на каждую):
  - Существующие решения (apps, services, products)
  - Реальные проблемы пользователей (Reddit, форумы, отзывы)
  - Заменители (alternatives, workarounds)
- **Опциональные линзы** (модель сама решает релевантность):
  - Региональные особенности (US/EU/Asia)
  - Свежие тренды
  - Failure stories
  - Нишевые варианты
  - Цифры / статистика

**Путь A для глубины:** 2-3 разных запроса на одну важную линзу (не один обзор) — даёт больше превью разных сайтов в сумме.

**Запросы — на английском** (лучший охват в Google/Reddit/Product Hunt). Конкретные, не общие: ❌ "renovation" → ✅ "renovation budget overrun reddit 2026".

**Маркеры уверенности per-пункт** в карточках:
- `[факт]` — общеизвестное, подтверждено
- `[типовой паттерн]` — устоявшаяся практика
- `[предположение]` — модель не уверена
- `[нужно проверить]` — гипотеза

**Confidence карточки целиком:** `search_snippet_supported` — НЕ `evidence_supported`, потому что сниппет это слабое evidence. Сильное evidence потребует deep research (отложено).

**Источники** возвращаются в виде аннотаций (`response.output_text.annotation.added`) — `{ title, url }`. Прикрепляются к карточке в поле `Card.sources[]`.

**Стоимость:** ~$0.50-0.70 за один Search Scan (по OpenAI pricing 2026: $10 за 1000 поисков + ~8K input tokens content на каждый поиск). За полный Reality Mapping всех 9 под-модулей: ~$0.90-1.20.

**Кнопка "🔬 Углубиться"** на конкретной карточке Search Scan — отложено на Шаг 3 (deep research через отдельный API).

**Если по линзе ничего не нашлось** — карточка явно пишет "по этой линзе свежей информации в поиске не обнаружено — пробел для ручного исследования". Пустоты должны быть видны.

## 4. Модель данных

### Card
```ts
interface Card {
  id: string;                    // crypto.randomUUID()
  number: number;                // #N в рамках этапа
  stageId: StageId;              // 'observation' | 'entity_mapping' | ...
  type: CardType;                // 'observation_item' | 'entity_dimension' | ...
  title: string;
  description: string;
  tags: string[];
  status: 'neutral' | 'interesting' | 'discarded';
  parentId: string | null;       // дочерняя карточка от другой
  createdAt: string;             // ISO
  // AI-генерация:
  metrics?: { novelty, strength, feasibility, testability };
  analysis?: string;             // объяснение метрик
  model?: string;                // 'gpt-5.5' / 'o4-mini' / etc.
  // Reality Mapping arch:
  confidence?: 'assumed' | 'user_provided' | 'observed' | 'evidence_supported';
  notes?: string;                // пользовательские заметки
  derivedFromIds?: string[];     // граф связей (данные сохраняются, UI отложен)
}
```

### Project
```ts
interface Project {
  id, title, frame, constraints[], criteria[];
  cards: Card[];
  activeStageId: StageId;
  selectedCardId: string | null;
  createdAt, updatedAt;
}
```

### localStorage
```
key: "creative_core_workbench_v1"
{ projects: Project[], activeProjectId: string | null }
```

## 5. Экраны и UX

### Project Picker
Пусто → форма New Project. Иначе список с `[Open]` / `[Delete]` + `[+ New Project]`.

### New / Edit Project Form
Поля: Title (required), Frame (required), Constraints[], Criteria[]. При сохранении нового — `activeStageId = 'observation'`, авто-старт генерации если есть API ключ.

### Workbench — 3 колонки

| Колонка | Ширина | Содержимое |
|---|---|---|
| **Stages** | ~200px | 18 stages в 4 модулях, иконки статусов, счётчик карточек |
| **Cards** | ~360px | Список карточек активного этапа, кнопки Генерировать / Думай ещё / Найти (Observation) / + Добавить |
| **Card Detail** | flex | Детали карточки: title, description, tags, метрики, status, model badge, action-кнопки |

### Header
Слева — название и краткое описание проекта. Справа:
- Git hash (кликабельный → ссылка на коммит)
- Селектор модели AI
- `🚀 Запустить всё` / `🔄 Перегенерировать` (auto-all)
- `⚡ AI готов` / `⚙ Ввести API ключ`
- Экспорт / Импорт JSON
- Проекты (обратно в Picker)

### Статусы этапов в левой колонке
- `✓` — есть ≥1 карточки `interesting`
- `●` — активный этап
- `·` — есть карточки, ни одной interesting
- `○` — карточек нет

### Auto-all (`🚀 Запустить всё`)
Последовательная генерация всех под-модулей по `order`: модель генерирует Observation Scan → Search Scan (через web_search) → Fundamental Dimensions → ... → Validation Plan. Каждый под-модуль получает все карточки предыдущих как контекст (через `derivedFromIds` модель указывает конкретные связи).

### Search Scan под-модуль (1.2)
Автоматический web search встроен в под-модуль 1.2 Search Scan через OpenAI Responses API + built-in `web_search` tool. Кнопка `🌐 Запустить поиск` запускает агентский цикл: модель сама формирует до 12 поисковых запросов, выполняет их, синтезирует найденное в карточки с источниками. Прогресс отображается живьём (текущий запрос, счётчик 1/12, фаза синтеза). Подробности в разделе 3.1 Search Scan архитектура.

## 6. Промпты (где живут)

Промпты для каждого этапа — в `src/utils/prompts.ts` в коде. Не копируются в ChatGPT отдельно — модель вызывается напрямую через OpenAI API из приложения.

**Структура промпта каждого этапа:**
- Контекст проекта (тема + ограничения + критерии)
- Карточки предыдущих этапов (все, ★-карточки помечены приоритетом)
- Описание задачи этапа
- Правила формата (JSON по строке, на русском)
- Поле `derived_from: ["id1", "id2"]` — связи с карточками предыдущих этапов

Эволюция промптов фиксируется в `prompts.ts`: текущие — наверху, legacy — внизу с пометкой `[LEGACY STAGE]`.

## 9. Анти-скоуп (для текущей итерации)

Это не «когда-нибудь». Явный запрет для текущей фазы — методологического теста.

❌ Не делаем:
- UI-визуализация графа связей `derivedFromIds` (данные сохраняем, визуализацию откладываем)
- Авторизация, шеринг проектов
- Drag-n-drop карточек
- Анимации, темы оформления
- Многоязычность интерфейса
- Undo / redo
- Поиск / фильтр карточек по тексту
- Сортировка карточек (только по `number`)
- Scoring агрегированный (метрики есть на карточке, но без общего рейтинга гипотез)

✅ Делаем (отступления от изначального анти-скоупа):
- **Linked cards** (`derivedFromIds`) — сделано как data-only, без UI. Модель указывает связи в JSON, они хранятся, передаются в промпты следующих этапов.
- **Notes** на карточках — поле есть в типе, UI откладываем
- **Web-поиск** — реализован через кнопку `🔍 Найти в интернете` (открывает Google, ручной ввод результатов в Search Notes)

## 10. Acceptance Criteria для MVP

1. Открываю приложение впервые → форма создания проекта
2. Создаю проект → попадаю в Workbench, активен 1.1 Observation Scan, авто-старт генерации всех под-модулей
3. Кнопка "🚀 Запустить всё" последовательно генерирует все 18 под-модулей по `order`
4. Под-модуль 1.2 Search Scan запускает web search автоматически через OpenAI Responses API
5. Метки `★ Отметить` / `✗ Отклонить` работают, видны на карточке и в списке
6. Создаю child-карточку с parent из предыдущего этапа → ссылка кликабельна
7. Перезагружаю страницу → состояние сохранено
8. Экспорт / импорт JSON работает
9. Auto-all с уже заполненным проектом → подтверждение "очистить и сгенерировать заново"
10. Старые проекты (legacy `definition` стейдж) открываются без крэша

## 12. Архитектура потоков данных

> Этот раздел — для понимания "как одно цепляется за другое". Если вносишь изменения в логику генерации или контекст между этапами — читай отсюда.

### 12.1 Поток методологии (поток данных между под-модулями)

Каждый под-модуль получает в промпте **все карточки всех предыдущих под-модулей** (по полю `order` в `STAGES`). Дополнительно модель в JSON указывает `derived_from: [id1, id2, ...]` — какие именно карточки она использовала как основу. Это образует граф зависимостей.

**Логические зависимости (что строится на чём):**

```
1.1 Observation Scan ──┐
                       ├──→ 1.3 Fundamental Dimensions ──→ 1.4 Obligatory vs Accidental ─────────┐
1.2 Search Scan 🌐 ────┘                                                                          │
                                                                                                   ↓
1.5 JTBD ←─── 1.3,1.4                                                                       2.1 Feature Challenge
1.6 Users ←── 1.5,1.3,1.4                                                                        │
1.7 Failure Modes ←── 1.3-1.6                                                                    │
1.8 Substitutes ←── 1.3-1.7                                                                      │
1.9 Boundary Cases ←── 1.3-1.8                                                                   │
                                                                                                  │
                              3.1 Friction Map ←── 1.4-1.9, 2.1 ─────────────────────────────────┤
                              3.2 Contradictions ←── 3.1 + 1.4                                    │
                              3.3 Cross-field ←── 3.2                                             │
                              3.4 Opportunity Tree ←── 3.1-3.3 + 2.1 ─────────────────────────────┤
                                                                                                  │
                              4.1 Hypothesis ←── 3.4 + 2.1                                        │
                              4.2 Critic ←── 4.1                                                  │
                              4.3 Shortlist ←── 4.1 + 4.2                                         │
                              4.4 Validation Plan ←── 4.3                                         │
```

**Ключевые мосты:**
- **1.4 → 2.1** — главный креативный bridge. Из "случайных" признаков рождаются трансформации.
- **2.1 + 3.4 → 4.1** — гипотезы строятся на пересечении challenge-ей и opportunity tree.

### 12.2 Три типа AI-вызовов

| Тип | Endpoint | Когда | Что возвращает |
|---|---|---|---|
| **`generateCardsStream`** | `POST /v1/chat/completions` (stream) | Любой обычный stage генерации карточек | JSON-объекты карточек по одной на строку через SSE delta |
| **`generateWithSearchStream`** | `POST /v1/responses` (stream + `tools: web_search`) | Только Search Scan (1.2) | Те же JSON-карточки + аннотации источников (`sources[]`) |
| **`askQuestionStream`** | `POST /v1/chat/completions` (stream) | "💬 Спросить" кнопка на карточке | Обычный человекочитаемый текст (не JSON), стримится в UI |

Все три используют:
- `stream: true` + `stream_options: { include_usage: true }` (для Chat Completions)
- AbortSignal параметр для прерывания
- LogContext параметр для записи в cost log
- Helpers `supportsCustomTemperature(model)` и `isReasoningModel(model)` для корректной передачи параметров под разные модели

### 12.3 Streaming pipeline (Chat Completions)

```
Browser
  │
  ├─→ fetch(/v1/chat/completions, { stream: true, stream_options: { include_usage: true } })
  │
  ↓
SSE stream:
  data: {"choices":[{"delta":{"content":"..."}}], ...}
  data: {"choices":[{"delta":{"content":"..."}}], ...}
  ...
  data: {"choices":[],"usage":{"prompt_tokens":N,"completion_tokens":M}}  ← финальный chunk с usage
  data: [DONE]
  │
  ↓
Парсер:
  - Накапливает delta в contentBuffer
  - При появлении \n в буфере — пытается распарсить строку как JSON-карточку
  - Если карточка валидна → onCard callback → ADD_CARD dispatch → ререндер CardsColumn
  - Параллельно копит totalContentChars (для fallback оценки токенов)
  │
  ↓
По окончанию stream:
  - flushAll() — финальный парсинг последней строки
  - Fallback оценка токенов если usage не пришёл (Math.ceil(len / 4))
  - appendCostEntry → запись в localStorage cost log
```

### 12.4 Streaming pipeline (Responses API + web_search)

Отличается типами SSE events:
- `response.output_text.delta` — текстовая дельта (аналог content)
- `response.web_search_call.in_progress` / `searching` — модель сейчас ищет, передаём query в onProgress
- `response.output_text.annotation.added` — пришёл источник (url+title), сохраняется в `pendingSources[]` и прикрепляется к следующей карточке
- `response.completed` — финал, usage и searchCount уходят в cost log

### 12.5 Auto-all loop (Workbench)

```typescript
for (let i = 0; i < STAGES.length; i++) {
  while (pauseRef.current && !stopRef.current) await sleep(250)  // пауза
  if (stopRef.current) break                                       // стоп
  
  if (stage уже заполнен) continue                                 // skip
  
  controller = new AbortController()
  autoAllAbortRef.current = controller
  
  const onCard = (gen) => {                                        // callback из stream
    const card = { id, number, stageId, ...gen, model, derivedFromIds, sources }
    generatedCards.push(card)                                      // локальный аккумулятор
    dispatch ADD_CARD                                              // глобальный store
  }
  
  if (stage.usesWebSearch) {
    await generateWithSearchStream(..., controller.signal)
  } else {
    await generateCardsStream(..., controller.signal)
  }
}
```

**Почему refs (`pauseRef`, `stopRef`) а не state:**  
async loop запущен один раз. React state-обновления не видны внутри уже-запущенной async function — она держит замыкание на старые значения. Refs обновляются мутабельно (через `pauseRef.current = ...`), поэтому проверка внутри loop видит актуальное значение.

**AbortController** — для мгновенного прерывания текущего fetch. Без него Stop ждал бы окончания текущего стейджа (до 1-2 минут).

### 12.6 State management

**Один useReducer** в `state/store.ts`. Все изменения через dispatch с типизированными actions:
- `CREATE_PROJECT` / `UPDATE_PROJECT` / `DELETE_PROJECT` / `SET_ACTIVE_PROJECT`
- `SET_ACTIVE_STAGE` / `SET_SELECTED_CARD`
- `ADD_CARD` / `UPDATE_CARD` / `DELETE_CARD`

**localStorage sync** — `useEffect` срабатывает на любое изменение state и сериализует в `localStorage[creative_core_workbench_v1]`. Чтение — один раз при инициализации reducer.

**Миграция** — в Workbench есть `useEffect` который при загрузке проекта проверяет валидность `activeStageId`. Если стейдж был переименован/удалён (как `definition` → `observation_scan`), переключает на `FIRST_STAGE_ID`.

### 12.7 Cost log

Хранится в `localStorage[cerebro_cost_log_v1]` — массив `CostEntry`, capped на 1000 последних записей.

**Источник цифр:**
1. Точные usage из OpenAI (если пришли в финальном SSE chunk) — приоритет
2. Fallback оценка: `Math.ceil(prompt.length / 4)` для input, `Math.ceil(totalContentChars / 4)` для output (1 токен ≈ 4 символа, неточно для русского)

**Цены** — в `utils/pricing.ts` таблица для каждой модели + per-call для web_search. Обновлять при изменении OpenAI pricing.

**UI** — `CostBadge` в хедере показывает total за сегодня (gray/amber/red по порогам). Клик → `CostModal` с breakdown по стейджу/модели/проекту, фильтр периода.

### 12.8 Q&A архитектура (кнопка "💬 Спросить")

Хранится на карточке как `Card.discussions[]` — массив `{ q, a, timestamp, model, tokensIn, tokensOut }`.

**Что в контексте при вопросе:**
- Сама карточка (title, description, теги, метрики, источники)
- Все её родители через `derivedFromIds` — полностью
- Все предыдущие Q&A этой беседы (как user/assistant в messages)
- Тема проекта + название этапа

**Поток:**
```
CardChatModal
  ↓ askQuestionStream(ctx, question, ...)
    ↓ buildCardChatMessages — собирает system + контекст + history + new question
    ↓ POST /v1/chat/completions с обычным system prompt ("отвечай простым языком, не JSON")
    ↓ stream → onDelta(chunk) → setStreamingAnswer(prev+chunk)
    ↓ финал → return { fullAnswer, tokensIn, tokensOut }
  ↓ dispatch UPDATE_CARD с новым discussions[]
  ↓ cost log пишется с пометкой "(вопрос к карточке)" в stageLabel
```

### 12.9 Промпты — структура

В `utils/prompts.ts`:
- `formatContext(project)` — тема + ограничения + критерии (всегда первым блоком)
- `formatCards(cards)` — все карточки контекста с маркером `★` для interesting
- `existingBlock(existing)` — "эти карточки уже сгенерированы, не повторяй" (для "Думай ещё")
- `FMT_RULES` — общий блок для всех стейджей: маркеры уверенности, формат списков, простой язык (rule 9), запреты на эссе, derived_from
- `FMT_RULES_METRICS` — `FMT_RULES` + блок метрик (для стейджей 4.1-4.4)
- `STAGE_PROMPTS: Record<StageId, PromptFn>` — конкретные промпты на каждый под-модуль

Каждый промпт возвращает строку вида:
```
{context}
{ancestors block — карточки предыдущих стейджей}
{existing block — если "Думай ещё"}
Stage: X.Y SOMETHING
...конкретные инструкции этапа...
{FMT_RULES или FMT_RULES_METRICS}
```

### 12.10 Что отложено архитектурно

- **Backend proxy** — клиент сейчас напрямую стучит в OpenAI с API ключом из localStorage. Промпты и ключ видны в DevTools Network. Для продуктизации нужен сервер.
- **Граф derivedFromIds в UI** — данные накапливаются, визуализации (стрелки, дерево, навигация) нет.
- **Умный фильтр контекста** — сейчас все карточки предыдущих стейджей идут полностью. Late-stage стейджи получают 100+ карточек = 30-40K input tokens. Идеи в Журнале (★-only для дальних предков / module-scoped / по derivedFromIds / summary).
- **Per-stage selector модели** — сейчас одна модель на весь auto-all. Логично: gpt-5.4-mini для простых стейджей (Observation, JTBD, Users, Substitutes), 5.5 для сложных (Feature Challenge, Hypothesis, Critic, Synthesis).
- **Prompt caching** — структура промптов сейчас не оптимизирована под автоматический кэш OpenAI (стабильное должно быть в начале, уникальное в конце).
- **Deep research** ("🔬 Углубиться" на карточке Search Scan) — переход в полный контент топ-сайтов через отдельный API (Perplexity / OpenAI deep-research / Tavily).
